## Understanding These Metrics

### Language Breakdown
- **YAML**: Configuration files, Kubernetes manifests, CI/CD workflows
- **Go**: Application code, tests, generated code
- **Markdown**: Documentation
- **Terraform**: Infrastructure as Code (IaC)
- **Protocol Buffers**: Domain message definitions
- **JSON**: API schemas, configuration
- **Dockerfile**: Container images

### Code Quality Indicators
- **Lines**: Total lines including code, comments, and blanks
- **Code**: Actual executable code
- **Comments**: Documentation and code comments
- **Blanks**: Empty lines for readability
- **Complexity**: Cyclomatic complexity (higher = more complex)

### Complexity Guidelines
- **< 10**: Simple, easy to test and maintain
- **10-20**: Moderate complexity, may benefit from refactoring
- **> 20**: High complexity, should be refactored into smaller functions

### Cost Estimates (COCOMO Model)
These are rough estimates assuming:
- Organic mode development (small-to-medium sized projects)
- Experienced team
- Industry-standard developer costs

**Note:** Actual development time depends on many factors including team experience, requirements clarity, and project complexity.

## Generating Metrics Manually

```bash
# View in terminal
task scc

# View per-file complexity
task scc-files

# Update this file
./scripts/generate-scc-report.sh
```

## Using Metrics for Code Quality

### Track Growth Over Time
```bash
git log -p docs/code-metrics.md
```

### Identify Refactoring Candidates
Run `task scc-files` and look for files with high complexity scores.

### Monitor Test Coverage
Compare Go test files to implementation files to ensure adequate coverage.

## Pre-commit Hook

This file is automatically updated before each commit. The hook:

1. Runs `scc` via Docker
2. Generates this report
3. Stages the updated file for commit

**Setup:**
```bash
pre-commit install
```

**Skip temporarily (not recommended):**
```bash
git commit --no-verify
```
